#  Лабораторна Робота 1

## 1. Обрана Предметна Область та Архітектурна Модель

### 1.1. Назва та Призначення ІС

| Параметр | Опис |
| :--- | :--- |
| **Назва** | **Image Processing API & Worker System** (Сервіс Асинхронної Обробки Зображень) |
| **Призначення** | Надання універсального **RESTful API** для прийому зображень від зовнішніх клієнтів, їх асинхронної обробки (зміна розміру, фільтри) та видачі готових результатів. Система спроєктована для роботи з великими обсягами завдань, де обробка може займати значний час. |

### 1.2. Вибір Архітектурної Моделі

* **Обрана Модель:** **Мікросервісна Архітектура**  із застосуванням **Черги Повідомлень**.
* **Обґрунтування:** Обробка зображень є ресурсомісткою та довготривалою операцією. Розділення системи на **API-Gateway** та **Worker-Service** дозволяє:
    * Клієнту отримувати **миттєву відповідь**, не чекаючи завершення обробки.
    * **Горизонтально масштабувати** Worker-сервіси незалежно від API-шлюзу, збільшуючи загальну пропускну здатність системи.

## 2. Визначення Користувачів та Точок Взаємодії

Система має двох ключових акторів, які взаємодіють з нею.

| Тип Користувача (Актор) | Роль | Основні Точки Взаємодії |
| :--- | :--- | :--- |
| **Клієнт (Client)** | Зовнішній застосунок (веб, мобільний) | **REST API (HTTP/HTTPS)** |
| **Внутрішній Worker-сервіс** | Автоматизований процес, що виконує обчислювальні завдання | **Message Queue (Redis/RabbitMQ)** та **Сховище Файлів** |

| Операція | Актор | Точка Взаємодії | Опис |
| :--- | :--- | :--- | :--- |
| **Завантажити зображення** | Клієнт | `POST /upload` | Завантаження файлу та отримання ID завдання. |
| **Перевірити статус** | Клієнт | `GET /status/{job_id}` | Перевірка статусу обробки в Кеші. |
| **Отримати завдання** | Worker | Message Queue | Споживання повідомлення із черги для початку обробки. |

## 3. Опис Нефункціональних Вимог (НФВ)

| Вимога | Показник (Metric) | Принцип Архітектури |
| :--- | :--- | :--- |
| **Масштабованість** | Підтримка до 1000 завдань на хвилину. | **Мікросервіси** дозволяють додавати більше екземплярів Worker-сервісу під час пікового навантаження. |
| **Надійність** | Uptime > 99.9%. Нульова втрата завдань. | **Асинхронна модель** та **Message Queue** гарантують, що завдання будуть оброблені, навіть якщо один Worker вийде з ладу. |
| **Продуктивність** | Час відповіді на `POST /upload` < 100 мс. | **Розділення відповідальності** (API лише приймає запит). |
| **Безпека** | Захист даних у транзиті. | Обов'язкове використання **HTTPS/TLS** для всіх зовнішніх API-запитів та аутентифікація через **JWT**. |
